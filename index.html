Problema que resuelve

Un usuario quiere resolver una tarea real, no solo recibir texto. El agente debe entender la solicitud, decidir qué ruta usar, consultar datos o documentos, ejecutar herramientas, validar la respuesta y escalar a una persona si la confianza es baja.

Solución planteada

Separar el sistema en bloques controlados: canal de entrada, webhook, orquestador, LLM, grafo de decisiones, herramientas, RAG, memoria, observabilidad, guardrails y revisión humana. Así la IA queda útil, trazable y más segura.

MCP

Conecta el LLM con herramientas externas mediante servidores especializados.

Usuario -> GPT -> MCP Server -> SQL Server -> Respuesta

Function / Tool Calling

El modelo decide llamar una función estructurada cuando necesita ejecutar una acción.

Saldo -> consultarSaldo() -> resultado -> respuesta

Chunking

Divide documentos grandes antes de crear embeddings. Puede ser fixed, recursive o semantic.

PDF 500 páginas -> chunks -> embeddings -> vector DB

Re-ranking

Ordena mejor los documentos recuperados antes de enviarlos al LLM.

Vector search -> 20 docs -> ReRank -> 3 mejores -> GPT

Tipos de prompt

  • Zero-shot: pedir una tarea sin ejemplos.
  • One-shot: dar un ejemplo para guiar el formato.
  • Few-shot: dar varios ejemplos para mejorar consistencia.
  • Chain of Thought: guiar razonamiento paso a paso cuando aplica.
  • Role prompting: asignar rol, contexto, restricciones y formato.

SaaS B2B AI-first: del problema a la solución

Este caso sirve para practicar cómo explicar una solución de IA de forma senior: no empezar por el modelo más llamativo, sino por el problema raíz, el riesgo, los datos disponibles y una arquitectura controlada.

Problema en términos simples

La empresa recibe información importante en PDFs, emails y formularios. Si esos datos se capturan manualmente, los procesos se vuelven lentos, propensos a errores y difíciles de auditar. Además, cualquier chatbot, reporte o automatización futura queda limitada por la mala calidad de la información de entrada.

Solución planteada

Crear un servicio de Document Intelligence dentro de la arquitectura de microservicios. La IA no modifica sistemas core directamente: extrae datos, entrega evidencia, calcula confianza y pasa por validadores antes de publicar resultados hacia CRM, ERP o el SaaS.

Ingesta -> almacenamiento seguro -> OCR/parsing -> clasificación -> extracción híbrida -> validación -> revisión humana -> publicación controlada -> auditoría

Arquitectura de alto nivel

  • Ingesta desde emails, APIs internas, carga manual o conectores.
  • Almacenamiento del original con tenant_id, usuario, origen, timestamp y permisos.
  • Preprocesamiento con OCR, limpieza, deduplicación y detección de tipo documental.
  • Cola o event bus para separar ingesta, extracción, validación y escritura.
  • Motor híbrido con reglas, templates, clasificadores y LLM cuando aporte valor.
  • Validación contra CRM, ERP, reglas de negocio y schemas versionados.
  • Bandeja humana para baja confianza, conflictos o campos sensibles.

Ejemplo real

Llega una factura PDF por email. El sistema extrae NIT, fecha, valor, moneda, cliente, contrato y orden de compra. Después valida cliente contra CRM, contrato contra ERP y reglas de negocio contra la plataforma SaaS.

Si los campos obligatorios tienen evidencia y confianza mayor o igual a 0.85, el registro queda listo para integración. Si falta evidencia o hay conflicto, pasa a revisión humana con el fragmento exacto que soporta cada dato.

Qué sí resolvería con IA

  • Interpretar documentos variables o sin plantilla fija.
  • Extraer campos desde texto libre y normalizar nombres, fechas o monedas.
  • Explicar al revisor por qué propone un valor.
  • Usar embeddings para buscar documentos similares o ejemplos históricos.
  • Usar RAG para consultar reglas por cliente, políticas internas o diccionarios de campos.

Qué no dejaría en manos de IA

  • Cálculos financieros críticos.
  • Permisos, segregación multi-tenant o escritura final en sistemas core.
  • Validaciones determinísticas entre ERP y CRM.
  • Decidir sin evidencia o sin umbral de confianza.
  • Ejecutar acciones sin auditoría, logs y control humano cuando aplique.

Cascada de decisión de modelos

1

Reglas / templates

Para documentos conocidos y campos fáciles. No usa LLM.

2

Modelo pequeño

Para clasificación, normalización simple o extracción de baja complejidad.

3

LLM grande

Para documentos variables, baja confianza o campos que requieren interpretación.

4

Revisión humana

Para campos críticos, evidencia incompleta o confianza menor al umbral.

MVP

  • Empezar con 2 o 3 documentos de alto volumen.
  • Crear dataset dorado con respuestas correctas por campo.
  • Usar OCR, parsing, reglas y templates para casos frecuentes.
  • Activar LLM solo cuando el template falle o el documento sea variable.
  • Integrar un único sistema destino y mantener revisión humana inicial.

Versión escalable

  • Separar ingesta, procesamiento, validación y escritura en servicios.
  • Usar colas, DLQ, reintentos idempotentes y autoscaling por cola.
  • Vector store por tenant, catálogo de schemas y evaluación automática.
  • Agregar caching, rate limits y selección dinámica de modelos.
  • Medir latencia, errores, coste, reintentos y revisión humana con OpenTelemetry.

Riesgos y controles

PIICifrado, tokenización, retención limitada y control por roles.
Multi-tenanttenant_id obligatorio en documentos, embeddings, logs y consultas.
Prompt injectionTratar documentos como datos no confiables y validar outputs.
TrazabilidadGuardar fuente, evidencia, modelo, prompt, schema, score y decisión.

Métricas de éxito

  • Precisión: más de 95% en campos críticos del dataset dorado.
  • Automatización: al menos 80% de documentos sin revisión humana después del MVP.
  • Tiempo: reducción del procesamiento por documento en 50%.
  • Calidad: reducción de errores manuales o retrabajo entre sistemas en 30%.
  • Coste: coste por documento y coste por campo exitoso dentro del umbral esperado.

Ejemplo 1: RAG

Asistente inteligente de productos, garantías y documentación técnica

Descripción del caso

Un asistente para asesores comerciales, servicio al cliente y equipos internos que responde preguntas sobre productos, fichas técnicas, garantías, políticas internas, compatibilidades, licenciamiento y documentación de fabricantes.

Problema que resuelve

Reduce el tiempo que los equipos invierten buscando información en PDFs, catálogos, manuales o documentos internos, y ayuda a entregar respuestas más consistentes.

Preguntas que podría responder

  • ¿Cuál es la política de garantía para portátiles HP?
  • ¿Qué características tiene esta referencia Lenovo?
  • ¿Qué condiciones aplican para la devolución de un producto?
  • ¿Qué diferencias hay entre dos modelos de impresoras Epson?
Documentos técnicos y comerciales -> extracción de texto -> fragmentos -> embeddings -> PostgreSQL + pgvector -> búsqueda semántica -> GPT responde con contexto.
Frase de entrevista: RAG permite que el modelo consulte información documental real antes de responder, reduciendo respuestas inventadas y mejorando la precisión.

Valor para la empresa

  • Centraliza conocimiento técnico y comercial.
  • Mejora la atención a clientes y canales.
  • Disminuye errores sobre garantías, productos o procesos.
Ejemplo 2: Text2SQL

Asistente para consultas comerciales, inventario y ventas

Descripción del caso

Un asistente para que líderes comerciales, gerentes y áreas operativas consulten ventas, inventario, productos, clientes, marcas, facturación o rotación de referencias usando lenguaje natural.

Problema que resuelve

Evita que los usuarios dependan siempre de analistas o reportes manuales para obtener información rápida de negocio, manteniendo validaciones de seguridad sobre las consultas.

Preguntas que podría responder

  • ¿Cuáles fueron los productos más vendidos este mes?
  • ¿Qué referencias Lenovo tienen bajo inventario?
  • ¿Qué clientes compraron licencias Microsoft en el último trimestre?
  • ¿Qué marcas tuvieron mayor rotación en los últimos 30 días?
Pregunta del usuario -> clasificación de intención -> generación SQL -> validación de seguridad -> ejecución en PostgreSQL o DWH -> GPT resume la respuesta. 
SELECT p.nombre_producto, p.referencia,
       SUM(v.cantidad) AS unidades_vendidas,
       SUM(v.valor_total) AS venta_total
FROM ventas v
JOIN productos p ON v.producto_id = p.id
JOIN marcas m ON p.marca_id = m.id
WHERE m.nombre = 'HP'
GROUP BY p.nombre_producto, p.referencia
ORDER BY unidades_vendidas DESC
LIMIT 10;
Frase de entrevista: Text2SQL permite que usuarios no técnicos consulten datos estructurados, siempre validando roles, permisos y consultas de solo lectura.

Controles clave

  • Permitir únicamente consultas SELECT.
  • Bloquear DELETE, UPDATE, INSERT, DROP o ALTER.
  • Registrar logs de pregunta, SQL generado y respuesta.

Resumen de arquitectura

Flask puede exponer la API backend, PostgreSQL puede funcionar como base relacional y vectorial con pgvector, SQLAlchemy puede manejar la conexión con datos, GPT puede generar respuestas, LangChain puede conectar los componentes y LangGraph puede orquestar decisiones, validaciones y rutas del flujo.

Idea clave: RAG consulta conocimiento documental, Text2SQL consulta datos estructurados y LangGraph decide qué ruta usar antes de entregar una respuesta segura.

Comparacion clave

Data Lake vs Lakehouse vs Data Warehouse

En entrevista puedes explicarlo como tres niveles de madurez para almacenar y consumir datos. El Data Lake guarda informacion cruda y flexible; el Lakehouse organiza ese lago con tablas, calidad y transacciones; el Data Warehouse deja datos modelados para SQL, BI y decisiones de negocio.

Data Lake

Almacena datos crudos o semiprocesados en archivos. Acepta CSV, JSON, Parquet, imagenes, logs y datos no estructurados.

  • Uso: aterrizar datos desde ERP, APIs, bases de datos o archivos.
  • Ventaja: flexible y economico para grandes volumenes.
  • Riesgo: si no hay gobierno, se convierte en un lago desordenado.

Lakehouse

Combina la flexibilidad del lake con tablas tipo Delta, calidad, versionamiento y procesamiento Spark.

  • Uso: capas Bronze, Silver y Gold para analitica, IA y BI.
  • Ventaja: sirve para datos crudos, limpios y analiticos en un mismo entorno.
  • Ejemplo: Fabric Lakehouse con OneLake y notebooks PySpark.

Data Warehouse

Base analitica estructurada para consultas SQL, modelos dimensionales, KPIs y reportes empresariales.

  • Uso: hechos, dimensiones, metricas oficiales y Power BI.
  • Ventaja: alto rendimiento para consultas de negocio repetibles.
  • Riesgo: menos flexible para datos crudos o no estructurados.

Migracion PostgreSQL -> Microsoft Fabric

Migracion analitica de PostgreSQL a Fabric con OneLake, Lakehouse, Warehouse y Power BI

Este ejemplo sirve para explicar una migracion realista: la empresa tiene PostgreSQL como base transaccional para ventas, clientes, productos, inventario y pagos, pero los reportes son lentos, hay consultas pesadas sobre produccion y cada area construye sus propios archivos. La solucion es llevar los datos a Microsoft Fabric para separar operacion de analitica, organizar capas Medallion y consumir indicadores desde Power BI.

PostgreSQL -> Fabric Data Factory / Mirroring -> OneLake -> Lakehouse Bronze -> PySpark Silver -> Warehouse Gold -> Modelo semantico -> Power BI / IA
1

Diagnostico

Inventariar tablas, volumen, llaves, indices, consultas criticas, ventanas de carga y dependencia de reportes.

2

Extraccion

Usar Copy Activity para cargas batch o Mirroring si se necesita replica casi en tiempo real hacia OneLake.

3

Bronze

Guardar tablas crudas en Lakehouse con fecha de carga, origen, hash de fila y control de lote.

4

Silver

Limpiar tipos de datos, nulos, duplicados, codigos, zonas horarias, monedas y reglas de calidad.

5

Gold

Crear hechos y dimensiones para ventas, clientes, productos, calendario, inventario y recaudo.

6

Consumo

Publicar modelo semantico con medidas DAX, RLS, refresh controlado y dashboards de Power BI.

Tablas origen en PostgreSQL

  • public.customers: clientes, segmento, ciudad, estado y fecha de registro.
  • public.products: SKU, categoria, marca, costo, precio y estado del producto.
  • public.orders: cabecera de venta, fecha, cliente, canal, moneda y vendedor.
  • public.order_items: detalle por producto, cantidad, descuento, impuesto y total.
  • public.inventory_movements: entradas, salidas, bodega, lote y fecha de movimiento.

Destino recomendado en Fabric

  • Lakehouse Bronze: replica cruda de PostgreSQL para trazabilidad y reprocesos.
  • Lakehouse Silver: tablas Delta limpias, tipadas y con reglas de negocio aplicadas.
  • Warehouse Gold: modelo dimensional optimizado para SQL, BI y usuarios de negocio.
  • Modelo semantico: medidas reutilizables y una unica version de indicadores.
  • Power BI: dashboards de ventas, inventario, margen, cartera y rotacion.

Consulta incremental desde PostgreSQL

SELECT
  o.id AS order_id,
  o.customer_id,
  o.order_date,
  o.updated_at,
  oi.product_id,
  oi.quantity,
  oi.unit_price,
  oi.discount,
  oi.tax,
  (oi.quantity * oi.unit_price) - oi.discount + oi.tax AS line_total
FROM public.orders o
JOIN public.order_items oi ON oi.order_id = o.id
WHERE o.updated_at >= :last_watermark
  AND o.updated_at <  :current_watermark;

Transformacion Silver con PySpark

from pyspark.sql import functions as F

bronze = spark.read.table("bronze_postgres.orders_raw")

silver = (
    bronze
    .dropDuplicates(["order_id", "product_id"])
    .withColumn("order_date", F.to_date("order_date"))
    .withColumn("line_total", F.col("line_total").cast("decimal(18,2)"))
    .withColumn("load_date", F.current_date())
)

silver.write.format("delta").mode("overwrite").saveAsTable("silver.orders_clean")

Validaciones obligatorias

  • Conteo de filas por tabla origen vs destino.
  • Sumas de control: ventas, impuestos, descuentos, cantidades e inventario.
  • Integridad referencial: ventas sin cliente, items sin producto, movimientos sin bodega.

Migracion PostgreSQL -> Microsoft Fabric

Migracion analitica de PostgreSQL a Fabric con OneLake, Lakehouse, Warehouse y Power BI

Este ejemplo sirve para explicar una migracion realista: la empresa tiene PostgreSQL como base transaccional para ventas, clientes, productos, inventario y pagos, pero los reportes son lentos, hay consultas pesadas sobre produccion y cada area construye sus propios archivos. La solucion es llevar los datos a Microsoft Fabric para separar operacion de analitica, organizar capas Medallion y consumir indicadores desde Power BI.

Tablas origen en PostgreSQL

  • public.customers: clientes, segmento, ciudad, estado y fecha de registro.
  • public.products: SKU, categoria, marca, costo, precio y estado del producto.
  • public.orders: cabecera de venta, fecha, cliente, canal, moneda y vendedor.
  • public.order_items: detalle por producto, cantidad, descuento, impuesto y total.
  • public.inventory_movements: entradas, salidas, bodega, lote y fecha de movimiento.

Destino recomendado en Fabric

  • Lakehouse Bronze: replica cruda de PostgreSQL para trazabilidad y reprocesos.
  • Lakehouse Silver: tablas Delta limpias, tipadas y con reglas de negocio aplicadas.
  • Warehouse Gold: modelo dimensional optimizado para SQL, BI y usuarios de negocio.
  • Modelo semantico: medidas reutilizables y una unica version de indicadores.
  • Power BI: dashboards de ventas, inventario, margen, cartera y rotacion.

Consulta incremental desde PostgreSQL

SELECT
  o.id AS order_id,
  o.customer_id,
  o.order_date,
  o.updated_at,
  oi.product_id,
  oi.quantity,
  oi.unit_price,
  oi.discount,
  oi.tax,
  (oi.quantity * oi.unit_price) - oi.discount + oi.tax AS line_total
FROM public.orders o
JOIN public.order_items oi ON oi.order_id = o.id
WHERE o.updated_at >= :last_watermark
  AND o.updated_at <  :current_watermark;

Transformacion Silver con PySpark

from pyspark.sql import functions as F

bronze = spark.read.table("bronze_postgres.orders_raw")

silver = (
    bronze
    .dropDuplicates(["order_id", "product_id"])
    .withColumn("order_date", F.to_date("order_date"))
    .withColumn("line_total", F.col("line_total").cast("decimal(18,2)"))
    .withColumn("load_date", F.current_date())
)

silver.write.format("delta").mode("overwrite").saveAsTable("silver.orders_clean")

Oracle + PySpark + Medallion

Extracción de datos Oracle y organización por capas Bronze, Silver y Gold

En un cargo de Python, PySpark y Big Data, Oracle normalmente se entiende como una fuente empresarial de datos: Oracle Database, Oracle ERP, Oracle EBS, Oracle Fusion, Oracle Financials o bases transaccionales. El objetivo no es solo consultar tablas, sino extraer información corporativa, llevarla a un entorno analítico y transformarla con trazabilidad y calidad.

Qué debes saber de Oracle

  • SQL Oracle: SELECT, JOIN, filtros, fechas, agregaciones y subconsultas.
  • PL/SQL: procedimientos, funciones, paquetes y lógica de negocio en base de datos.
  • Tablas transaccionales: facturas, clientes, pagos, productos e inventario.
  • Vistas: consultas guardadas que exponen datos organizados para consumo.
  • Índices y performance: ayudan a evitar extracciones pesadas sobre producción.
  • JDBC: método común para conectar Spark o PySpark con Oracle.

Cómo entra PySpark

PySpark puede leer tablas o consultas desde Oracle usando JDBC, cargar los datos como DataFrames distribuidos, aplicar limpieza, validaciones y reglas de negocio, y escribir resultados en un Data Lake, Lakehouse, Data Warehouse o capa analítica. 

df_oracle = spark.read \
    .format("jdbc") \
    .option("url", "jdbc:oracle:thin:@host:1521/service") \
    .option("dbtable", "SCHEMA.TABLA_CLIENTES") \
    .option("user", "usuario") \
    .option("password", "password") \
    .option("driver", "oracle.jdbc.driver.OracleDriver") \
    .load()

ventas_mensuales_gold, cartera_cliente_gold, kpi_recaudo_gold
Oracle ERP / Oracle Database -> JDBC -> PySpark DataFrames -> Bronze -> Silver -> Gold -> Power BI / analítica / IA

Transformaciones típicas

  • Eliminar duplicados y validar nulos.
  • Normalizar nombres de columnas y convertir fechas.
  • Homologar códigos de producto, cliente o moneda.
  • Cruzar facturas, pagos, clientes, productos e inventario.
  • Construir KPIs de ventas, cartera, recaudo o disponibilidad.

Respuesta fuerte para entrevista

Oracle puede funcionar como una fuente transaccional principal. Desde PySpark extraería datos con JDBC, los llevaría a Bronze conservando la estructura original, aplicaría limpieza y reglas de negocio en Silver, y construiría tablas Gold para reportería, KPIs o modelos analíticos.

Resumen para entrevista

La arquitectura integra fuentes como ERP, APIs y bases de datos; procesa la información con Python, PySpark y SQL; y deja datos confiables para automatización, analítica, BI o soluciones con IA.

Idea clave: Python aporta flexibilidad, PySpark escala el procesamiento, SQL valida los datos y BI convierte la información en decisiones.
